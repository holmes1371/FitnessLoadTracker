# #32 тАФ Recent syncs row: deflated count + per-item error text

**Closed:** 2026-05-26
**Branch:** `claude/pensive-lamport-2647e7`

## What was wrong

Every BG row in the Recent syncs list was showing "1 activities" with the тЪа warning icon, while the per-item list in `syncResults` correctly collapsed to the empty-state message. Two distinct issues stacked on the same row:

1. **Misleading count.** `SyncLogEntry.activitiesProcessed` was set to raw `items.count` in `SyncOrchestrator`'s `defer`, so the 24h overlap re-fetching the prior day's ride (already stamped `.skippedAlreadyHasEffort`) inflated the count to 1. #24's `displayItems` filter de-noised the per-item UI list but didn't touch the persisted log count тАФ the row count and the displayed-list count had drifted out of sync.

2. **Opaque тЪа warning.** `SyncLogEntry` stored `perItemErrors: Int` but discarded the actual error string from `.error(msg)` statuses. The тЪа icon was firing on BG rows but Tom couldn't see *why* without an Xcode debugger attached. Likely culprit (still to confirm from the next failure surfaced post-fix): HK queries throwing when the BG task fires on a locked device, but we were flying blind without the message.

## What changed

- **`SyncLogEntry.firstItemError: String?`** тАФ captured from the first `.error(msg)` status in items at log time, in both `syncActivities` and `syncSingleActivity` defer blocks. Optional so existing persisted entries decode cleanly via `decodeIfPresent`.

- **Deflated `activitiesProcessed`** тАФ both defer blocks now compute `processed = items.filter { not .skippedAlreadyHasEffort }.count`. Other skip kinds (`.skippedNoMatch`, `.skippedNoSufferScore`, `.skippedMultipleMatches`) and errors still count тАФ they're signal, not noise.

- **Recent syncs row UI** тАФ when the тЪа fires, `errorSummary ?? firstItemError` renders in caption2 + .red below the HStack so Tom can read the actual error string on the phone without an Xcode session.

- **`SyncLogTests`** тАФ round-trip the new field; explicit legacy-JSON decode test (`legacyJSONDecodes`) that constructs the pre-#32 JSON shape and confirms `SyncLog.recent()` returns it with `firstItemError == nil`. Without that test, an Optional Codable property could silently default to non-nil or тАФ worse тАФ fail the whole array decode and wipe Tom's history on upgrade.

## Decisions made and rejected alternatives

- **`.skippedAlreadyHasEffort` is the only skip kind excluded from the count.** Other skips mean the activity was fetched but couldn't be processed for a non-routine reason; Tom wants to see that on the row. Skipped-already-has-effort is the pure-overlap re-fetch noise that #24 identified and is uniquely safe to silence.

- **Did not add a separate `skippedAsAlreadyHasEffort: Int` field.** The audit trail of "how many skips happened" lives in-memory in `items[]` during the sync; persisting a second counter just for the historical view bloats the entry and wasn't asked for. If Tom wants this later, it's a one-line add.

- **Did not refactor the two defer blocks into a helper.** They've already been parallel since #24 and the new logic is 3 lines repeated; extracting now would be scope creep against the karpathy "surgical changes" rule. If a third call site appears, that's the right moment to consolidate.

- **`firstItemError` not `firstError`.** The field name pairs with `perItemErrors: Int`, both prefixed `perItem` / `firstItem` so a future reader knows this is the *per-item* error, not the top-level `errorSummary`. The two are kept distinct because `errorSummary` is "the sync itself failed (network/OAuth)" and `firstItemError` is "the sync ran, this one activity errored" тАФ different remediation paths.

- **Inline error text, not tap-to-expand.** Tap-to-expand would require local view state, a gesture, and probably an animation тАФ overkill for a debug surface that's expected to be empty most of the time. Inline caption2 is cheap and readable.

## What surfaced during verification
