# #2 — Phase 0: Register Strava API application

## What was built

User-only setup task. Tom registered a Strava API application at https://www.strava.com/settings/api and captured the `client_id` + `client_secret` to his password manager. No agent code, no repo commits as part of this issue. Closed 2026-05-24 after Tom verified the application is visible in Strava settings, creds are stored securely outside the repo, and the authorization callback domain is set to `localhost`.

## Decisions worth preserving

- **Callback domain set to `localhost`** during registration, not the GitHub Pages URL or any deployed domain. The iOS app's OAuth flow (#4) will use `ASWebAuthenticationSession` with a custom URL scheme; `localhost` is the conventional "I'm a native app, not a web app" value Strava accepts. Strava lets you update the callback domain after registration, so #4 can swap to the actual custom URL scheme without re-registering the app.

- **Credentials NOT in GitHub Secrets.** GitHub Secrets exist for GitHub Actions workflows; we have none and don't plan to. The original "GitHub Actions cron → JSON" architecture was rejected (see `design/architecture.md`), so a Secrets bucket has no consumer. Storing creds there would add an attack surface (org members, runners) for zero benefit. The creds live in Tom's password manager and will be injected into the iOS build via a gitignored local `.env` at `ios/.env` on the Mac, picked up by an Xcode build phase. At runtime the `client_secret` is embedded in the app binary — acceptable for a personal sideloaded app because the worst case (someone extracting the binary) still requires Tom's iPhone Keychain refresh token to actually read his Strava data. If we ever ship beyond Tom's own phone, that calculus changes; revisit then.

- **Phase 0 was filed as its own issue rather than folded into #4.** It's a manual user action with no code, which would have muddied #4's "agent-built" scope. Separate issue means clean audit trail and a clear manual-verification checklist Tom personally ticks. Worth the extra issue.

## Trigger and design history

The whole pipeline assumes the iOS app can talk to the Strava API on Tom's behalf — which requires registering an OAuth application with Strava. That requirement falls out of the rejected-paths analysis earlier in the same session: Apple Shortcuts can't write `workoutEffortScore` to existing workouts (`Log Health Sample` doesn't expose the type and even if it did, no Shortcut-equivalent for `relateWorkoutEffortSample`), and existing third-party apps (HealthFit / RunGap / SyncMyTracks) sync workouts and HR but never write effort from external sources. With those off the table, a native iOS app talking to Strava became the locked architecture, making API credentials a hard prerequisite for any Strava-side work.

## Follow-ups

- **#4 will need to update Strava's callback domain or add the custom URL scheme** used by `ASWebAuthenticationSession`. Strava's API settings page allows post-hoc edits.
- If BG Refresh in #5 proves unreliable enough that a server-side cron becomes necessary, *that's* when GitHub Secrets would enter the picture — for the cron's runtime env, not the iOS app. Defer until measured.
