# #22 — Migrate writeWorkout to HKWorkoutBuilder

## What was built

Replaced `HealthKitManager.writeWorkout`'s deprecated `HKWorkout(activityType:start:end:duration:totalEnergyBurned:totalDistance:metadata:)` init with `HKWorkoutBuilder`. Both persistent deprecation warnings from #17 are gone — the iOS-17 init warning at `HealthKitManager.swift` and the iOS-18 `HKWorkout.totalEnergyBurned` accessor warning at `HealthKitManagerTests.swift`. Clean build, no warnings.

The pure-builder split that #17 introduced for testability stayed, just reshaped: `WorkoutBuild` (which held an `HKWorkout` instance + HR samples) → `WorkoutBlueprint` (pure data fields — activityType, start, end, duration, totalDistance, totalEnergyBurned, metadata, HR samples). `buildWorkoutData` → `buildBlueprint`. The static function no longer constructs an `HKWorkout` at all; `writeWorkout` consumes the blueprint and drives `HKWorkoutBuilder` (begin / addMetadata / addSamples / end / finish).

Landed in one commit on branch `claude/epic-nightingale-3f8ef5`:
- `0150681` — full migration: WorkoutBlueprint struct, buildBlueprint static func, distanceQuantityType helper, expanded shareTypes, HKWorkoutBuilder-driven writeWorkout, new builderReturnedNil error case, all tests rewritten to assert on blueprint fields, new test for the mapping helper.

Verified by deploying the post-migration build + resyncing: a mix of "Effort N" pills (matched path on existing HK twins) for cycling and the newly-mapped strength training (see #26), plus "Created + Effort N" pills for any Strava-only outdoor rides not yet in HK (create path through the new builder). No regressions on the matched-path side.

## Decisions worth preserving

### `WorkoutBlueprint` over `WorkoutBuild`

`HKWorkoutBuilder` requires a real `HKHealthStore` at construction and computes totals from `addSamples` calls rather than direct values. The "build then save" pattern from #17 — where `buildWorkoutData` returned an `HKWorkout` instance for tests to inspect, and `writeWorkout` saved that instance — doesn't survive the migration: there's no equivalent way to construct an in-memory `HKWorkout` from a configuration without going through the builder, and the builder won't operate without a healthStore.

Resolution: keep the pure-builder split, but the pure side no longer produces an `HKWorkout`. The blueprint holds the inputs to the builder — activityType, time bounds, distance and energy quantities, metadata, HR samples — and tests assert on those fields directly. `writeWorkout` is the only code path that touches the builder + healthStore. The architecture is structurally the same as #17 (pure constructor + impure save), the seam just moved one level earlier.

This is the right shape going forward: the test boundary is "the inputs we feed the builder," not "the workout object the builder returns." If `HKWorkoutBuilder`'s API evolves further, the blueprint stays valid; only `writeWorkout`'s body changes.

### `distanceQuantityType(for:)` defaults to walking/running

`HKWorkoutBuilder.addSamples` needs cumulative distance samples typed correctly per activity — `.distanceCycling` for cycling, `.distanceSwimming` for swimming, `.distanceWalkingRunning` for everything else that involves moving on land. The five activity types `Matching` currently emits (`.cycling`, `.running`, `.walking`, `.hiking`, `.swimming`) all have a defined mapping; cycling and swimming get explicit cases, the rest fall through `default` to `.distanceWalkingRunning`.

The default-branch fallback is deliberate: if `Matching` ever gains a new mapping (and #26 added `.traditionalStrengthTraining` later this same session), the new activity type routes through the default without a compile error. That's a property worth keeping for forward extension — adding `case "Crossfit": return .crossTraining` to `Matching` doesn't have to be paired with a `distanceQuantityType` update to compile.

The cost: for activities that don't have meaningful distance (strength training, yoga, etc.), the default writes a `.distanceWalkingRunning` cumulative sample of 0 meters. Semantically wrong but invisible in Health.app and otherwise harmless. See *Gotchas* for when this might bite and the cheap fix if it ever does.

### Cumulative samples span the full workout, not per-second

For cumulative quantity types (distance, active energy), the builder sums all added samples to produce the workout's total. The simplest correct shape: one sample per cumulative quantity, spanning the full `startDate..endDate` range, carrying the total value. That's what we do — one `.distance{Cycling|WalkingRunning|Swimming}` sample with `detail.distance`, one `.activeEnergyBurned` sample with `detail.calories`, both with the workout's start and end dates.

The alternative — per-second resampling to look like Apple Watch native data — would have added 3600+ samples per hour-long workout for distance and energy, with no UI difference. The Health.app's workout-detail view doesn't break out cumulative quantities into per-second graphs the way it does for HR. Sticking with one cumulative sample each is the simpler, correct shape.

### `HKWorkoutConfiguration.locationType` left unset, preserving #17 parity

The deprecated init had no slot for `locationType`. #17 didn't set it. The new code path has a slot (`HKWorkoutConfiguration.locationType` accepts `.indoor` / `.outdoor` / `.unknown`), and Strava distinguishes `Ride` (outdoor) from `VirtualRide` (indoor) at the sport-type level, so the data exists upstream.

Deliberately not setting it as part of this migration. Adding it changes behavior beyond the migration's scope: outdoor cycling workouts would start carrying `.outdoor`, which can affect how Watch surfaces them and how Health.app groups them. That's a feature decision for its own issue, not a side effect of the migration. Default unset = "unknown" preserves parity with #17.

### `WriteWorkoutError.builderReturnedNil` typed but unexpected

`HKWorkoutBuilder.finishWorkout()` returns `HKWorkout?`. The docs are vague on when it returns nil — typically only after an `endCollection` failure or builder misuse, neither of which we expect on the success path. But the API signature is `Optional`, and silently force-unwrapping or skipping the case would let a misconfigured builder produce a no-op save with no error surfaced.

Typed `WriteWorkoutError.builderReturnedNil` as the throwable for this branch. If it ever fires, the error message ("HKWorkoutBuilder.finishWorkout returned nil — workout not saved.") is clear about what happened, and the orchestrator's existing error handling propagates it into the activity-row UI as a failed sync — no silent data loss.

### The `@available` removal is a no-op

The issue's scope item 5 ("Remove the `@available(iOS, deprecated: 17.0)` annotation introduced in #17") was carried over from when the issue was filed mid-#17, before the `296eb0e` → `5648eb0` reversal landed. By the time #22 started, the annotation was already gone. Documented in the issue body and ticked as "no-op (already absent)" to keep the audit trail honest — if anyone reads #22 cold and looks for the removal commit, they'd otherwise spend time hunting for an absence.

The underlying lesson #17 captured (`@available` propagates the warning to callers, doesn't silence it) is preserved in `completed/17.md`.

## Gotchas

- **Non-distance-bearing activities get a `.distanceWalkingRunning` sample with 0 meters via the default branch.** `distanceQuantityType(for:)`'s switch is exhaustive only over distance-bearing types; everything else (currently just `.traditionalStrengthTraining` after #26, but any future `.yoga` / `.functionalStrengthTraining` / etc. would also fall through) gets a 0-meter walking distance sample on its app-authored workout. Invisible in Health.app's workout detail and harmless to other queries, but technically wrong data. If a future debug pass surfaces a 0-meter walking sample on a strength workout and it bothers us, the fix is a 4-line guard in `writeWorkout`: don't add the distance sample at all when the activityType isn't in the distance-bearing set. Not fixing pre-emptively because (a) the path requires a Strava-authored strength activity with no Watch recording, which is rare on Tom's setup, and (b) the wrong-but-harmless sample doesn't propagate to any visible surface.

- **`shareTypes` expansion triggers a fresh auth prompt.** Adding `.distanceCycling`, `.distanceWalkingRunning`, `.distanceSwimming`, and `.activeEnergyBurned` to the share set means iOS will re-prompt for HK write authorization on the first run after the upgrade. Tom approved the prompt during verification; future fresh installs will see all seven write types in the prompt instead of three. No way around this — `HKWorkoutBuilder.addSamples` won't accept a sample of a type you don't have write auth for.

- **HKWorkoutBuilder is documented for live-recording flows; we're using it for after-the-fact reconstruction.** The Apple docs talk about builders for in-progress workouts where you start collection at workout-start, add samples as they arrive in real time, and end collection at workout-end. Our use is "the workout already happened, we're recreating it from Strava data" — all timestamps are in the past, all samples added in one shot. This works because the builder is implementation-agnostic about whether `beginCollection(at:)` is called before or after the actual moment; it just needs samples to fall within the begin..end range. Just an FYI for anyone reading the code and wondering why the flow doesn't look like Apple's example.

- **Test coverage for the writeWorkout side is still manual.** The blueprint side has full unit-test coverage (suite renamed to `HealthKitManagerBuildBlueprintTests` with all four #17 cases preserved + a new test for `distanceQuantityType(for:)`). The builder-driven save side has none — same constraint as #17's HKWorkout-instance side and `writeEffort`: no precedent for mocking `HKWorkoutBuilder` or `HKHealthStore`, and that test infrastructure is #23's job. Verified manually via the post-deploy resync, same convention as #17.

## What's NOT in here

- **`HKWorkoutConfiguration.locationType`.** Slot exists, data is in Strava, but adding it changes Apple's surfacing behavior. Its own issue if/when we want outdoor/indoor distinction in Health.app.
- **Distance-sample-type guard for non-distance-bearing activities.** Documented in *Gotchas* as a 4-line fix; deferred until the create path for strength training (or yoga, or whatever) actually fires and the 0-meter sample shows up somewhere visible.
- **Orchestrator integration tests.** Still #23. The migration didn't change the `writeWorkout(detail:streams:) -> HKWorkout` signature, so the orchestrator integration is structurally untouched.
- **Per-second distance / energy sample resampling.** Considered, rejected — see *Decisions worth preserving / Cumulative samples span the full workout*.

## Future tweaks

- If the create-path 0-meter distance-walking-running sample on a strength workout ever surfaces somewhere visible, drop the distance sample entirely for activities not in `{.cycling, .running, .walking, .hiking, .swimming}`. Cheap, mechanical fix.
- If we want `.indoor` / `.outdoor` distinction in Health.app workout cards, add a `Matching.locationType(forStravaSportType:)` helper and set `HKWorkoutConfiguration.locationType` in `writeWorkout`. Strava's sport_type taxonomy (e.g., `Ride` vs `VirtualRide`) carries enough info to drive this without additional API calls.
- If we ever migrate to a different HK ingestion strategy (e.g., a server-side push or an `HKWorkoutSession` flow), the `WorkoutBlueprint` shape stays valid; only `writeWorkout`'s body needs to change.
