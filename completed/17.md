# #17 — Single-writer HK pipeline: app creates HKWorkouts for Strava-only activities

## What was built

The app is now the single source of truth for Strava-activities in HealthKit. Tom disabled Strava's native HK sync (one-time, per the walkthrough in the issue body), and the sync orchestrator now branches per-activity:

- **HK twin exists** (Peloton ride, Apple-Watch-authored ride/hike, etc.) → attach `workoutEffortScore` via the unchanged matched path. Recent syncs shows the green "Effort N" pill.
- **No HK twin, sport_type is mapped** → fetch `/activities/{id}` + `/activities/{id}/streams?keys=heartrate,time` from Strava, build an `HKWorkout` with full metadata + per-second HR samples, save it, attach effort. Recent syncs shows the green "Created + Effort N" pill (the new `ItemStatus.writtenAsNew(effort:)` case).
- **No HK twin, sport_type unmapped** (Yoga, WeightTraining, etc.) → `.skippedNoMatch`, no orphan workout created.

Replaces #12's Peloton-over-Strava tiebreaker entirely — with Strava→HK off, the Strava bundle stops appearing in HK candidates at all, so the 2-candidate special case became dead code.

Landed in eleven commits on branch `claude/epic-lalande-c636a9`:

- `10250cc` — Plan revision: broader metadata scope, `moving_time` drives duration.
- `3bbb8d7` — Step 1: HK share-auth expanded for `HKWorkoutType` + `HKQuantityType(.heartRate)`.
- `14a24e7` — Step 2: `StravaClient.fetchActivityDetail` + `StravaActivityDetail` decode.
- `e929060` — Step 3: `StravaClient.fetchActivityStreams` + `StravaStreams` decode.
- `296eb0e` — Step 4: `HealthKitManager.writeWorkout` + pure-builder `buildWorkoutData` + tests.
- `5648eb0` — Follow-up: remove misplaced `@available` annotation that propagated deprecation noise to every caller.
- `bde756d` — Step 5: orchestrator routes the no-match-but-mappable branch through writeWorkout.
- `f99699b` — Step 6: rip Peloton-over-Strava tiebreaker + `WorkoutCandidate.bundleID`.
- `46c365f` — Targeted single-activity sync (`syncSingleActivity` + debug UI) for spot-check before backfill.
- `2b68fe1` — Backfill capability: paginate `fetchActivities`, add `syncBackfill`, one-time UI button.
- `4fc1577` — Remove the backfill button (kept `syncBackfill` + pagination for #24's reuse).

Verification: targeted-test on the 10/2/2025 ride first, then full backfill of 179 activities from 2024-10-10 onward. Mix of pill statuses matched the expected behavior (`Effort N` for twinned, `Created + Effort N` for Strava-only outdoor rides, `Already has effort` for the targeted-test ride on re-sync, `No score` / `No match` per skip rules).

## Decisions worth preserving

### Single-writer architecture supersedes the tiebreaker

#12 treated the symptom (always pick Peloton when both candidates are present), not the cause (Strava's native HK sync writing the duplicate in the first place). The pivot to "disable Strava→HK + become the writer for Strava-only activities" makes the duplicate impossible by construction rather than recoverable-after-the-fact. The tiebreaker code is gone — `Matching.swift` is back to a clean "1 match → matched, 2+ → multipleMatches" shape, no special-case for two specific bundle IDs. If a future feature ever needs the bundle IDs again, they're in git history and #12's post-mortem.

### `moving_time` drives `HKWorkout.duration`, not `elapsed_time`

Strava distinguishes the two: `elapsed_time` is wall-clock including auto-pauses; `moving_time` is time actually working. Tom's training-load reasoning is built on moving time (the number that drives the suffer score), so that's what should drive `HKWorkout.duration` and feed `HKWorkout.endDate = startDate + moving_time`. `elapsed_time` is still captured as custom metadata (`com.holmes.fitnessloadtracker.elapsedTime`) so the wall-clock figure isn't lost — just doesn't drive the primary duration field.

### `start_date` (UTC) drives `HKWorkout.startDate`, not `start_date_local`

Strava's `start_date_local` is a misleadingly Z-suffixed local-wall-clock string ("2026-05-20T13:00:00Z" representing 1pm local, not 1pm UTC). Parsing it as ISO8601 produces a `Date` that's the local-wall-clock time interpreted as a UTC moment — which then displays shifted by the user's tz offset when HK renders it in device tz.

Using `start_date` (true UTC) avoids the shift entirely. HK stores the absolute UTC moment, renders in the device's tz; on Tom's Eastern-locked phone, a 1pm Eastern ride displays as "1pm" with zero adjustment code. The only failure mode is if Tom travels to a different tz and views the same workout on his Eastern phone — he'd see the original ride time translated to Eastern, not preserved as the local-at-workout time. We accepted that trade-off as the simpler path; if it ever bites, the recovery is to construct a `Date` explicitly from `start_date_local`'s components interpreted in a known fixed tz.

### Pure-builder split because `HKWorkoutBuilder` is incompatible with unit tests

`HKWorkout(activityType:start:end:duration:totalEnergyBurned:totalDistance:metadata:)` is deprecated in iOS 17 in favor of `HKWorkoutBuilder`. The builder requires a real `HKHealthStore` at construction time and computes totals from sample objects rather than direct values — both incompatible with the project's no-HK-mocking convention. Used the deprecated init instead, split the build into a pure static `buildWorkoutData(detail:streams:) -> WorkoutBuild` function that returns an `HKWorkout` + `[HKQuantitySample]` array without saving. Tests assert on those outputs; the save side is manual-verified per existing convention (same as `writeEffort` has been since #4).

Migration to `HKWorkoutBuilder` (with a `WorkoutBlueprint` struct + `addSamples` for distance/energy) tracked in #22. Two persistent deprecation warnings remain — both at `HealthKitManager.swift:~124` (the init) and `HealthKitManagerTests.swift:~46` (`HKWorkout.totalEnergyBurned` accessor, deprecated in iOS 18). Both vanish with the #22 migration.

### Pull broader metadata than HK can currently render

Apple's first-class metadata keys (`HKMetadataKeyAverageSpeed`, `HKMetadataKeyMaximumSpeed`, `HKMetadataKeyElevationAscended`) render in the Fitness app and on the Watch. Custom keys under `com.holmes.fitnessloadtracker.*` (HR avg/max, elapsed_time, cycling power avg/max/weighted/kilojoules, cadence, deviceName, stravaActivityId, stravaWorkoutType) don't surface in any Apple UI but are recoverable via `HKAnchoredObjectQuery` if a future view wants them.

The marginal cost per workout: one decoded field + one dict entry. The alternative — realizing later we want a field and re-syncing the entire Strava history — is much more expensive. Specifically `stravaActivityId` is load-bearing for any future "trace this HK workout back to its Strava source" feature; not pulling it now would be a hard-to-recover-from loss.

Deliberately skipped: weather metadata (Strava populates inconsistently), `gear_id` (Strava-internal reference, useless without a separate lookup pass), splits/segments/laps as `HKWorkoutEvent`s (mapping is involved and there's no concrete use case).

### Sport-type guard preserved on the no-match path

`Matching.findMatch` returns `.noMatch` for two distinct reasons: (a) sport_type isn't in the HK activity-type map, (b) sport_type is mapped but no HK candidate matches within tolerance. The orchestrator's `.noMatch` branch re-checks `Matching.hkActivityType` and falls back to `.skippedNoMatch` for (a) so we don't try to create an HKWorkout for an unknown sport. Without this guard, unmapped sport types (Yoga, etc.) would fall into the create path and throw at `buildWorkoutData`'s `unmappedSportType` precondition.

The matching `case .noMatch:` and the buildWorkoutData precondition are two independent guards against the same failure. Belt-and-suspenders is fine here — both are cheap.

### Targeted-test capability added pre-backfill

Tom's call: spot-check one known-good ride before letting the create path loose on years of history. The `syncSingleActivity(id:source:healthKit:)` orchestrator method + debug UI block lets you paste a Strava activity ID and flow it through the full `process()` switch — same routing as a normal sync, just over a one-element items array. The `StravaActivityDetail.asSummary` projection collapses a detail-endpoint response down to the `StravaActivity` shape `Matching` consumes, so the targeted path doesn't duplicate orchestration logic. `preFetchedDetail` optional on `process()` avoids a redundant detail fetch when the caller already has one.

This capability stays in the app post-#17. It's useful debug surface for any future spot-checking and costs ~30 lines of UI.

### Pagination in `fetchActivities` fixes a latent bug

Pre-#17, `fetchActivities` fetched one page of `per_page=30`. At Tom's volume of 5–10 rides/week, a 30-day window can exceed 30 activities and silently drop the oldest — they'd never get effort attached. The pagination fix (loop with `per_page=200` until a partial page signals the tail) shouldn't change normal-sync behavior for Tom but closes that latent gap and makes multi-month backfill possible in finite round-trips.

The 19-month backfill landed in 2–3 pages and ~250-500 activities. Strava's rate limit (100/15min, 1000/day) was comfortably under-budget.

### HR samples: instantaneous timing (start == end), not 1-second windows

Apple Watch native HR samples use `start == end` for per-beat readings; the Fitness app's HR graph interpolates between point samples to draw the curve. Matching that convention keeps the app-authored HR graph indistinguishable from a Watch-native one. The other option — `end = start + 1s` to imply a 1-second sampling window — would have produced a subtly different rendering. Not worth the divergence from Apple's pattern.

### Orchestrator integration tests punted to #23

`SyncOrchestrator` has zero unit tests pre-#17 because the codebase has no precedent for mocking `StravaClient` (real network) or `HealthKitManager` (real HK). Adding that infrastructure means extracting `StravaClientProtocol` + `HealthKitManagerProtocol`, threading them through DI, writing fakes — ~150+ lines just for testability, with payoff beyond #17 (it'd benefit every future orchestrator change).

Decision: not in #17. Building blocks (`MatchingTests`, `HealthKitManagerBuildWorkoutDataTests`, `StravaActivityDetailTests`, `StravaStreamsTests`) are unit-tested; end-to-end behavior covered by manual verification, which the 179-activity backfill exercised at scale. Test-infra work lives in #23 as its own focused issue.

## Gotchas

- **Strava → HK native sync must stay off.** The pipeline assumes Strava no longer writes to HK directly. If Tom (or someone else) re-enables "Write to Health" in the Strava iOS app, two things happen: (1) Strava-authored duplicates start appearing in HK again, and (2) the orchestrator's matching window will see them as candidates, so `findMatch` will return `.matched` for what should be the create path — the existing Strava-authored HKWorkout gets the effort attached instead of an app-authored one. End result is similar to pre-#17, but the pipeline's invariants are violated. The walkthrough in the issue body documents how to confirm.

- **Two persistent deprecation warnings until #22 lands.** `HealthKitManager.swift:~124` (HKWorkout init deprecated iOS 17) and `HealthKitManagerTests.swift:~46` (`totalEnergyBurned` accessor deprecated iOS 18). Both are at the actual deprecated-API call sites — annotated in commit messages and a comment above the init. Don't try to silence them with `@available`; that propagates the warning instead of suppressing it (the attempt and reversal are in `296eb0e` and `5648eb0`).

- **HK source-app icon may lag.** Apps showing icons in Health.app's source-attribution UI (like Strava, Peloton) rely on iOS caching the app's icon per bundle ID. Newly-installed apps' icons sometimes don't appear immediately. Recovery (in order): force-quit Health.app, restart phone, reinstall the app. Not a configuration issue on our side — the asset catalog is set up correctly (`Contents.json` references a 1024×1024 master image; iOS 18 auto-generates the smaller variants).

- **`StravaActivityDetail.startDateLocal` parses as UTC.** The `.iso8601` strategy interprets the Z suffix at face value, so the resulting `Date` is the local-wall-clock time *as if* it were UTC. Currently unused at write time (`start_date` is what feeds `HKWorkout.startDate`). If a future caller wants the true local wall-clock, they need to re-interpret this field's components in a known fixed tz rather than trusting the parsed `Date`.

- **`StravaActivityDetail.sufferScore` was added late.** Initially the detail endpoint's decode struct didn't include suffer_score because the matched path gets it from the `StravaActivity` summary. The targeted-test path discovered the gap — it fetches detail and converts to summary, and the conversion needs `sufferScore`. Added in `46c365f`. The summary endpoint and detail endpoint both return suffer_score; both decode paths now include it.

- **Per-second HR fidelity is high but not Watch-native.** Strava's `/streams` endpoint returns 1Hz HR samples. A 60-minute ride is 3600 samples saved in a single batched `save([HKQuantitySample])` call. That's fine for HK and matches Apple Watch's cadence. If Strava ever degrades to a lower-resolution stream (e.g., for activities that didn't record HR continuously), the empty-data check in `buildWorkoutData` skips HR sample creation cleanly — no decode crash, no empty-array save.

## What's NOT in here

- **`HKWorkoutBuilder` migration** → #22.
- **Orchestrator-level integration tests** → #23.
- **Incremental sync** (track `lastSuccessfulSyncAt`, only fetch new activities, "No new activities" empty state) → #24.
- **UI warning when Strava→HK is still on** (would detect `com.strava.stravaride` in HK candidates) — listed in design's `Out (deferred)` section. Possible future signal-quality feature.
- **Backfill before 2024-10-10.** Tom's chosen start. The `syncBackfill` method takes any `after: Date`, so older windows are possible; just no UI to trigger them today (would re-add a button or pass via the upcoming incremental-sync UX).
- **Non-HR streams** (power-second, cadence-second, distance-time, altitude). Available from the same `/streams` endpoint by adding more `keys`. Tom's primary HR need is covered; other streams stay deferred per design's `Out` section.

## Future tweaks

- **If sport-type mismatch between Strava and HK becomes a recurring duplicate source**, `Matching` can map a single Strava sport type to a *set* of acceptable HK candidate types, or relax the activityType filter when other tolerances are tight. Examples Tom might hit: "Walk" vs `.hiking` for a Strava-walk that Apple recorded as a hike, "Run" vs `.running` vs `.cardio` for treadmill workouts.
- **Matching window** is currently ±60s on start time + duration. If post-backfill Tom sees some duplicates that should have matched, widening to ±90s or ±120s is a one-line change in `Matching.toleranceSeconds`.
- **Custom-metadata view in the app.** All the `com.holmes.fitnessloadtracker.*` keys we now write are queryable via `HKAnchoredObjectQuery`. A future in-app workout-detail view could surface cycling power, cadence, deviceName, etc. — data Apple Fitness doesn't show but we own.
- **`stravaActivityId` as a dedupe key.** If we ever need to detect "did we already create an app-authored HKWorkout for this Strava activity," the metadata key lets us look it up directly rather than re-doing the start/duration matching. Useful for any future "re-run create for activity X" flow that needs idempotency.
