# #28 — Silence Swift 6 main-actor Equatable conformance warnings

**Closed:** 2026-05-26
**Branch:** `claude/28-equatable-nonisolated`

## What was wrong

`⌘U` produced 16 warnings of the form "Main actor-isolated conformance of 'X' to 'Equatable' cannot be used in nonisolated context; this is an error in the Swift 6 language mode." Affected types: `SyncLogEntry` (1 warning), `FailureNotifier.Decision` (7), `MatchResult` (7). Pre-existing; surfaced during #24 verification.

Root cause: the project sets `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` in both Debug and Release configs ([project.pbxproj](ios/FitnessLoadTracker/FitnessLoadTracker.xcodeproj/project.pbxproj) lines 370 + 399), so every type is implicitly `@MainActor`. Synthesized `Equatable` conformances inherit that isolation, and the test suite — which Swift Testing runs from a nonisolated context — can't use them. Tests still passed under Swift 5 mode, but the warnings would become hard errors under Swift 6.

## What changed

Three-keyword edit: added `nonisolated` to each offending type declaration. The three types are pure value types with no main-actor semantics, so opting them out of the project default is correct and minimal.

- `nonisolated struct SyncLogEntry` in `SyncLog.swift`
- `nonisolated enum Decision` (inside `FailureNotifier`) in `FailureNotifier.swift`
- `nonisolated enum MatchResult` in `Matching.swift`

No test changes — the existing assertions just compile cleanly now.

## Decisions made and rejected alternatives

- **Per-type `nonisolated` vs global setting flip.** The issue offered both. I chose per-type because flipping `SWIFT_DEFAULT_ACTOR_ISOLATION` to `nonisolated` would also un-isolate every `@Observable` class — `HealthKitManager`, `StravaConnection`, `SyncOrchestrator` — that's bound to SwiftUI state and needs to stay on the main actor. The fix would have to expand to adding `@MainActor` to each of those (and possibly others), which is much more surface than #28 asked for. The surgical fix is enough; the global flip is its own day if Tom wants it.

- **Did not touch the other Equatable types** (`StravaActivity`, `StravaActivityDetail`, `StravaStreams`, `StravaStreams.Stream`, `WorkoutCandidate`, `HealthKitManager.Status`, `StravaConnection.State`, `SyncOrchestrator.ItemStatus`, `SyncOrchestrator.Item`). They aren't currently warning because no test compares them from a nonisolated context. Some (the value types — `StravaActivity` etc.) probably *should* be `nonisolated` too on principle, but karpathy "surgical changes" wins — fix what's broken, leave the rest. Future warnings will surface them and we'll fix them then. The ones nested in `@Observable` classes (`Status`, `State`, `ItemStatus`, `Item`) genuinely need to follow the parent's isolation, so they stay as they are.

- **`nonisolated` on the type declaration, not on a separate extension.** Two valid syntaxes exist: `nonisolated struct Foo: Equatable {}` or `extension Foo: @retroactive Equatable {} where Foo is nonisolated`. The former is one keyword on one line; the latter splits the declaration. Picked the simpler form.

## What surfaced during verification

Nothing — Tom ran `⌘U`, all tests passed, the 16 Equatable warnings dropped to 0. Clean fix.
