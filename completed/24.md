# #24 — Incremental sync via SyncCheckpoint + SyncWindow

**Closed:** 2026-05-25
**Branch:** `claude/magical-almeida-fd9a65`

## What was wrong

`SyncOrchestrator.syncRecentActivities` always fetched a 30-day window from Strava regardless of when the last successful sync had run. Most activities in a routine Sync ended up `.skippedAlreadyHasEffort` — but only after a paginated Strava list call plus an HK twin lookup per activity. Wasteful at best, and obviously broken UX: tapping "Sync now" 30 seconds after the last sync should not re-walk a month of history.

## What changed

Two new types and a small orchestrator-side wiring:

- **`SyncCheckpoint`** — `UserDefaults`-backed `Date?` keyed at `dev.holmes.fitnessloadtracker.lastSuccessfulSyncAt`. Modeled on `SyncLog` (static `save`/`load`/`clear`, injectable defaults for tests). Deliberately separate from `SyncLog`: `SyncLog` records every attempt (success + failure, capped at 10 entries); `SyncCheckpoint` is the durable watermark of the last *clean* run, with no cap and no failure mixing. Reusing SyncLog's newest entry would have entangled "did we record a sync?" with "did it succeed?" — and would have lost the watermark past 10 entries.

- **`SyncWindow.resolveAfterDate(...)`** — pure helper, extracted so the window logic is testable without standing up the orchestrator. Three cases:
  - Fresh checkpoint (≤7 days old): `after = last − 24h overlap`. The overlap absorbs activities that post to Strava hours after they end (a known Strava-side behavior).
  - Nil checkpoint (fresh install): `after = now − 30 days`. Matches the prior default; #17's backfill button covers deeper history.
  - Stale checkpoint (>7 days old): same 30-day fallback. Protects against bugs or outages that leave the checkpoint frozen.

- **`SyncOrchestrator`** — `syncRecentActivities` drops the `daysBack:` parameter (no caller wanted to override it), computes `after` via `SyncWindow.resolveAfterDate(lastSuccessfulSyncAt: SyncCheckpoint.load())`, and writes `SyncCheckpoint.save(finishedAt)` in the `defer` only when `errorMessage == nil`. Per-item errors don't block the checkpoint advance — the next sync's overlap covers them. `syncSingleActivity` records `lastSyncFinishedAt` for the UI but never advances the checkpoint; a targeted one-off fetch shouldn't move the full-window watermark.

- **`ContentView`** — new `lastSyncFinishedAt: Date?` on the orchestrator gates the empty-state message ("No new activities since [last sync time]") so it shows only after a sync has finished this session, never on cold open. A follow-up commit added a `displayItems` filter that hides `.skippedAlreadyHasEffort` items from the per-item list, so a no-op Sync collapses to the empty-state message even when the 24h overlap re-fetches recent activities. Other skip kinds (`.skippedNoMatch`, `.skippedNoSufferScore`, `.skippedMultipleMatches`) and errors stay visible — they're signal, not noise. Recent syncs still shows the full processed count for audit.

## Decisions made and rejected alternatives

- **Storage**: chose a dedicated `SyncCheckpoint` over reusing `SyncLog`'s newest-entry timestamp. The issue body offered both; the alternative loses ground when SyncLog rolls past 10 entries and confuses success with attempt.
- **First-install default**: 30 days, not 90 days, not a UI prompt. Matches prior behavior, no UX change, #17's backfill button is the right escape hatch for older history.
- **Stale threshold**: 7 days, as suggested. A single constant in `SyncWindow.defaultStaleAfter`; easy to revisit.
- **Targeted single-activity sync**: deliberately does NOT advance the checkpoint. Confused this on first pass; corrected after considering the failure mode (debug single-activity → next full Sync skips everything before today).

## What surfaced during verification

- **First Sync**: ran clean on the 30-day fresh-install window, attached effort to recent activities, advanced the checkpoint. ✓
- **Second Sync immediately after**: 24h overlap re-fetched the activity from an hour earlier, stamped it `.skippedAlreadyHasEffort` (as expected at the data layer) but the UI was showing the row as if something happened. Fixed in `f16bb75` by filtering `.skippedAlreadyHasEffort` out of the displayed list. Second Sync now collapses to "No new activities since [first sync time]" — even with the overlap re-fetch — which is what Tom wanted.
- **Pre-existing Swift 6 warnings**: 16 warnings under `⌘U` about main-actor-isolated `Equatable` conformances on `SyncLogEntry` / `FailureNotifier.Decision` / `MatchResult`. All pre-date this branch; tests pass; filed as #28 for separate cleanup. Likely root cause is the project's "Default Actor Isolation" build setting being set to `MainActor`.
