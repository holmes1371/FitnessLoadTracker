# #26 — Map Strava WeightTraining to HK .traditionalStrengthTraining

## What was built

Added one case to `Matching.hkActivityType(forStravaSportType:)`:

```swift
case "WeightTraining":
    return .traditionalStrengthTraining
```

Apple Watch records strength sessions as `HKWorkoutActivityType.traditionalStrengthTraining`. Strava's sport_type taxonomy labels the same activity `WeightTraining` (distinct from the activity *name*, which is "Morning/Afternoon/Night Weight Training"). Without this mapping, a Watch-authored strength session and the paired Strava activity stayed un-matched in the orchestrator's `findMatch` window, so the effort score never got attached.

Landed in one commit on branch `claude/epic-nightingale-3f8ef5`:
- `7a593f5` — one-line addition to `Matching.swift`, one row added to the parametrized `sportTypeMapping` test in `MatchingTests`.

Surfaced during #22's post-migration verification — the most recent strength training activity was a convenient way to exercise the matched path on a non-cycling activity type while also confirming the migrated `writeWorkout` didn't regress cycling syncs. Two birds, one resync. Verified: green "Effort N" pill on the strength activity in Recent Syncs; the corresponding `.traditionalStrengthTraining` workout in Health.app showed the attached effort score.

## Decisions worth preserving

### One sport type at a time, not a speculative batch

Strava's sport_type taxonomy has many more values we don't yet map: `Crossfit`, `Workout`, `Elliptical`, `Yoga`, `Pilates`, `RockClimbing`, etc. The temptation while in `Matching.hkActivityType` is to add half a dozen at once. Rejected: each mapping is a one-line change that's easy to add when the activity type first shows up in Tom's sync history. Adding mappings speculatively means owning the create-path correctness for activity types Tom may never actually log — and `distanceQuantityType(for:)`'s default branch is correct for cycling-adjacent / walking-adjacent activities but wrong for things like yoga or rock climbing where distance is meaningless.

The convention going forward: when Tom logs a new Strava activity that doesn't get an effort pill, file a one-line issue, add the mapping, ship it. Cheap loop.

## Gotchas

- **Create path for Strava-only strength training writes a 0-meter walking distance sample.** Covered in detail in `completed/22.md`. Summary: if a `WeightTraining` Strava activity ever lacks a Watch-authored HK twin, `writeWorkout` will run, and `distanceQuantityType(for: .traditionalStrengthTraining)` falls through to the `default` → `.distanceWalkingRunning`. The 0-meter walking distance sample is invisible in Health.app and harmless to other queries, but technically wrong. Unlikely to fire on Tom's setup (he records strength on the Watch). If it ever surfaces, the fix is a 4-line guard in `writeWorkout` — defer until then.

- **The matching test suite uses a parametrized table.** Adding a new sport_type means one row in `MatchingTests.sportTypeMapping`. The test names are auto-generated from the arguments, so the new row is `Strava sport types map to expected HKWorkoutActivityType ("WeightTraining", traditionalStrengthTraining)` — easy to see in test output. Pattern carries forward for future mappings.

## What's NOT in here

- Mappings for any other Strava sport_type. Each gets its own issue if/when it matters.
- The non-distance-bearing-activity guard for the create path. Deferred per *Gotchas*.
