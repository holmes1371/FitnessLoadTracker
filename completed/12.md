# #12 — Resolve Peloton/Strava duplicate HKWorkouts during effort sync

## What was built

A Peloton-over-Strava tiebreaker in `Matching.findMatch`. When the sync sees exactly two HKWorkouts matching a Strava activity and one is `com.Peloton.PelotonApp` + the other is `com.strava.stravaride`, the effort write routes to the Peloton-sourced workout instead of skipping the activity entirely with `skippedMultipleMatches`. Any other multi-match shape (two Peloton, two Strava, 3+, or mixed third-party) still falls through to `skippedMultipleMatches`. Tom keeps deleting the Strava-sourced duplicate in Health.app manually; the sync no longer leaves those rides without an effort score.

## Decisions worth preserving

### Original scope was "delete the Strava duplicate" — that path is dead

Indoor Peloton rides hit Apple Health twice: once from the Peloton iOS app (carries the Watch HR captured during the ride), once from Strava when Peloton's Strava sync round-trips the ride back. Initial framing: have the agent delete the Strava-sourced HK duplicate during the same sync that writes Relative Effort.

POC on Tom's actual device resolved the gating question: **`HKHealthStore.delete([workout])` on a Strava-authored workout returns `com.apple.healthkit 3 — Failed to find some objects for deletion`**. This is consistent with Apple's documented restriction that an app can only delete samples it has authored. The app having `HKWorkoutType` in its `toShare` set is not sufficient — share authorization grants write-our-own access, not blanket delete authority over data from other sources. HealthKit silently strips foreign-source samples from the delete request and reports the resulting empty set as "couldn't find."

A second angle Tom asked about — *mutate* the Strava workout's stats to zero, rather than delete — also dead-ended for two compounding reasons: (1) `HKWorkout` is immutable post-creation (no API to change `startDate`/`endDate`/`totalEnergyBurned`/`totalDistance`), so "edit" would mean delete-and-replace, hitting the same restriction; (2) Strava-authored per-sample stats (`activeEnergyBurned`, `distanceCycling`, etc.) fall under the same authorship rule. Negative compensating samples don't work either — HealthKit rejects negative values for most quantity types. The HealthKit security model is consistent: read across all (with permission), write our own, delete/mutate our own. There's no API path to "reduce or alter another app's data."

### Pivot from "delete the duplicate" to "tiebreaker for the effort write"

Once deletion was out, the underlying need shrank from "remove the Strava duplicate from HealthKit" to "make sure the effort lands on the *right* one of the two." Tom keeps doing manual cleanup in Health.app; the sync's only job is to write effort to the Peloton workout and leave the Strava workout alone.

Implementation lives in `Matching.findMatch`. When `matches.count == 2` and the two `bundleID`s are exactly `{com.Peloton.PelotonApp, com.strava.stravaride}`, return the Peloton match's index. Anything else: still `.multipleMatches`. Surgical — the rule only fires on the exact pair we know about.

### `WorkoutCandidate` grows `bundleID` instead of moving the rule into `SyncOrchestrator`

Two places to put the tiebreaker:
1. **In `Matching`** — keeps the matching logic complete and testable on plain structs. Adds `bundleID: String` to `WorkoutCandidate` (`SyncOrchestrator` populates from `sourceRevision.source.bundleIdentifier`). Tests cover all cases without an HKHealthStore.
2. **In `SyncOrchestrator`** — touches the matched HKWorkout directly. Less testable (mocking HKWorkout is painful), and splits the matching policy across two files.

Chose (1). Matches the existing pattern from `completed/4.md`: "`WorkoutCandidate` struct separates matching from `HKWorkout`." The plain-struct discipline keeps paying off.

### Bundle IDs (confirmed via the POC probe, repo-searchable)

- Peloton iOS app: `com.Peloton.PelotonApp`
- Strava iOS app: `com.strava.stravaride`

Hardcoded as private constants on `Matching`. If a future feature needs to recognize either app's data from HealthKit, grep these.

### Why the rule is scoped to *exactly 2* matches with *exactly this pair*

Considered broadening to "prefer Peloton over anything" or "always prefer the first non-Strava source." Rejected — both would silently change behavior for cases we haven't characterized (two Peloton workouts on the same Strava activity is genuinely ambiguous; mixing in a third-party HealthKit writer in the future would be a different problem). Conservative scope: tiebreaker fires only on the exact 2-element set of bundle IDs we confirmed via the POC. Three-way mixes and same-source duplicates stay `.multipleMatches` so they're still surfaced to the UI for inspection.

### "Set effort 7" debug button removed in same branch

The button was scaffolding from early HealthKit-write validation (#4a) that let Tom verify the `workoutEffortScore` + `relateWorkoutEffortSample` write path on a known workout without needing a Strava connection. End-to-end sync via #4b/#4c now exercises the same write path, so the button is dead weight. Dropped it along with the `setEffortSevenOnMostRecentWorkout()` method and the private `mostRecentWorkout()` helper that had no other callers. `Status` enum kept intact — `.failure` still surfaces auth errors; `.working`/`.success` cases are now unreached but cost nothing.

## Process notes

### POC discipline: probe-then-pivot beat speculative scoping

The original issue scope had four downstream sub-tasks gated on the POC ("Confirm Peloton-on-Strava detection signal", "Extend sync pipeline: pair-and-prune", etc.). Treating the POC as a true gate — build the minimal probe, run on real data, *then* decide — saved building any of the gated work against a fundamentally broken premise. The probe took one round-trip (push branch → Tom builds in Xcode against the worktree's `xcodeproj` → reports bundle IDs + error code) and resolved the gating question in under an hour.

The probe scaffolding (`recentWorkoutsWithSource`, `deleteWorkout`, `ProbeRow`, the purple "POC: Probe HealthKit sources" button) was reverted in the same branch (`d36284d`) once it had served its purpose. The committed history retains `f4a23f2` as the breadcrumb for what was tried.

### Verification-on-the-working-branch flow formalized in CLAUDE.md

This issue was the first where Tom tested directly on the agent's working branch via Xcode pointed at the worktree's `xcodeproj`, rather than waiting for the PR to merge into main and then verifying. The pattern made the POC round-trip cheap (no PR overhead) and surfaced the gating answer in a fraction of the previous flow's wall-clock time.

Codified in CLAUDE.md session discipline in the same PR (commit landing alongside `completed/12.md`):
- New bullet *Verification on the working branch* describing the Xcode-at-worktree pattern.
- *PR timing* trigger word shifts from "ready for testing" to "raise the PR" / "ready to merge" since testing now happens before the PR opens.
- *Close-keywords in PR bodies* — previously forbidden, now allowed and expected. Merge = verified = close.
- *Closed items archive to `completed/<NN>.md`* — moved from "right when the issue closes" to "right before the closing PR opens," since the post-mortem now rides the feature PR onto main rather than being committed via a parent-worktree-main bypass.
- *Sessions that don't produce a feature PR* — the parent-worktree-main bypass is retained as a fallback for research-only / pause-mid-feature sessions, but it's no longer the standard path.
- "In Testing" board column now legacy/unused under this model. Left on the board, but no issues route through it.

### Manual verification waived on close-out (low-risk change, no current test fixture)

Tom had manually deleted every Strava duplicate from the last ~60 days before this issue was filed. That left no in-the-wild test fixture for the duplicate-pair path; the unit tests in `MatchingTests` cover the matching logic deterministically, but on-device verification of the full sync flow against a real Peloton+Strava duplicate pair has to wait for the next time he rides the Peloton and a duplicate appears organically.

Tom chose to merge on test-suite trust rather than hold the issue open for weeks waiting for a fixture. Risk profile is low: the worst case if the tiebreaker fires wrong is the sync reverts to today's behavior (`skippedMultipleMatches`), which is the same UX as not having the fix at all. The new flow's *Verification on the working branch* bullet explicitly carves out this case — "(or Tom explicitly waives it on a low-risk change he's content to ship on test-suite trust)."

The Manual verification checklist on the issue body stayed unticked at close; that's by design and matches the waiver. If the tiebreaker misfires in the wild, file a new bug — don't reopen #12.
