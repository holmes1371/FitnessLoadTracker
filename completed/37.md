# #37 — Sync equivalent cycling distance for indoor/Peloton rides

**Closed:** 2026-05-28
**Branch:** `claude/hungry-ardinghelli-6f4913`

## What was wrong

Peloton indoor rides reach HealthKit via the Peloton app with time, calories, and Watch HR — but **no cycling distance**. Strava (mirroring Peloton) carries the equivalent distance, so the data existed upstream; it just never landed in Health. Goal: get that distance into Health on every sync, keeping Peloton as the workout writer so the Apple Watch HR is preserved.

## The governing constraint

HealthKit only lets an app modify samples it authored (verified in #12). The Peloton app owns the indoor-ride workout, so we **cannot** attach a distance sample to it — there's no cross-source "relate distance" API (unlike `relateWorkoutEffortSample`, which is why effort already works on Peloton workouts). Everything below is a consequence of working around that wall.

## Approach evolution (three takes)

The naive read of the request — "add distance to the workout" — is impossible, so we iterated on where the distance *can* live, with Tom steering on real-device behavior:

1. **Standalone `distanceCycling` sample (shipped first, then abandoned).** Author an unattached sample spanning the ride. It lands in the **Cycling Distance data type** (Browse → Cycling Distance) and its trend chart — but is **invisible** to both the individual workout card *and* the per-workout "Cycling Workouts → Distance" rollup, because those only surface distance attached to a workout. Tom verified on-device that the number was there in the data type but nowhere he actually looks. Not good enough.

2. **Take over the workout (rejected).** Disable Peloton→HK; let the app recreate the full ride via the existing `.noMatch` create path (which already writes distance). Distance would show everywhere, app-owned. Rejected by Tom: he wants the Watch HR that the Peloton app carries, and doesn't want app-owned workouts that vanish on uninstall for his primary ride type.

3. **Near-zero-duration parallel "proxy" workout (shipped).** Tom's idea: author our *own* 1-second cycling workout carrying just the distance, sibling to the Peloton workout. Because it's a workout we own, its distance rolls into "Cycling Workouts → Distance". The **1-second** duration is load-bearing: it keeps the proxy's contribution to the cycling TIME rollup and to Apple's Exercise-minutes ring negligible. Accepted tradeoff: it shows as a "0-minute, X-mile" entry in the workout list.

## How the ongoing feature works

- On a **matched** cycling workout, `enrichDistanceIfNeeded` writes a proxy only when `DistanceEnrichment.shouldWrite` passes: cycling, Strava distance > 0, the matched workout has **no native distance**, and we haven't already written one. The native-distance guard (`workoutHasNativeCyclingDistance`, reads `workout.statistics(for: .distanceCycling)`) means outdoor rides — and any Peloton ride that *does* carry native distance — are skipped, so no duplicate distance is ever created.
- Proxies are stamped with `stravaActivityId` + a `distanceProxy` metadata flag, and **excluded from match candidates** (`workouts(in:).filter { !isDistanceProxy($0) }`). Without that, a proxy would be a second cycling workout near the ride's start and could trip `multipleMatches` or steal the effort write from the real Peloton workout.
- Effort and distance writes are **decoupled** — a ride synced before #37 already has effort but no distance, so the effort-dedup must not short-circuit the distance write. Both are idempotent.
- `distance` was added to the `StravaActivity` summary model, so the matched path needs **no extra Strava API call** (distance comes free in the `/athlete/activities` feed).

## The duplicate-workout bug (found mid-flight) and its fix

A 2-year backfill button was built to migrate history. Running it surfaced a **latent create-path bug**: Tom ended up with two of each previously-synced outdoor ride. Root cause, two halves:

1. The `.noMatch` create path had **no dedup by Strava activity id** — it created unconditionally.
2. An app-created workout can't reliably re-match itself: `buildBlueprint` stores `moving_time` as the workout's duration, but `Matching` compares against `elapsed_time`. For outdoor rides with stops the gap exceeds the 60s tolerance → `noMatch` → a fresh duplicate every re-sync. (Indoor rides were immune: they match the Peloton-written workout, never the create path.) The backfill just made a slow latent leak instant and large.

Confirmed on-device: both copies were authored by FitnessLoadTracker (not a device), which pinned it to the create path.

**Permanent fix:** in `.noMatch`, before creating, look for a workout we already authored for this Strava id (it's present in the fetched candidates, just rejected by `Matching`) and route it through the shared `handleMatchedWorkout` (ensure effort, no-op distance) instead of creating a second copy.

**One-time cleanup:** a `removeDuplicateAuthoredWorkouts(since:)` plus a "Remove duplicate workouts" button swept the existing mess — grouped app-authored workouts by `(stravaActivityId, isProxy)`, kept one of each, deleted the rest **plus their associated distance/energy/HR samples** (deleting only the workout shell would leave orphaned samples still double-counting). Hard safety: only ever touched workouts authored by this app *and* stamped with our Strava id, so device-written rides and any single copy were untouchable.

## Decisions made and rejected alternatives

- **Detection by "lacks native distance," not by "is Peloton."** No hardcoded device/app name. Any cycling workout missing its own distance gets enriched; anything with distance is left alone. Simpler and self-correcting.
- **1-second proxy, not full-duration.** Full duration would double the cycling TIME rollup and Exercise minutes, and (because its duration would match the ride) could itself trip the matcher. 1s sidesteps both. Proxies are *also* explicitly filtered from candidates so we don't lean on duration alone.
- **Retired the migration tools after use.** Once the create-path dedup landed (making re-syncs safe) and the existing duplicates were swept, the "Backfill 2 years" and "Remove duplicate workouts" buttons + their backing code were deleted, and the cleanup-only read permissions (`distanceWalkingRunning`/`Swimming`, `activeEnergyBurned`, `heartRate`) reverted. `distanceCycling` read stays — the native-distance guard needs it. UI is back to just "Sync now."
- **Did not fix the `moving_time` vs `elapsed_time` matching mismatch directly.** The create-path id-dedup makes it moot for our own creations, which is where the duplication bit. Broadening `Matching`'s tolerance or storing both durations is a separate design question, left out of scope.

## Testing

- `DistanceEnrichmentTests` covers the pure gate's truth table (indoor writes; native distance / already-written / zero-distance / non-cycling all skip).
- `StravaActivity` / `StravaActivityDetail` decode tests pick up the new `distance` field; `HealthKitManager` share/read-set tests assert `distanceCycling` read.
- The HealthKit-touching methods (proxy write, dedup query, cleanup) follow the project's existing pattern of not being unit-tested directly — the pure gate carries the decision logic. Full XCTest pack green throughout.

Final state — regular Sync only — left for his last build. Manual-verification boxes on the issue stay his to tick.
