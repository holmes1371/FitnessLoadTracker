# #30 — Empty-state shows the previous sync's time, not the just-completed one

**Closed:** 2026-05-25
**Branch:** `claude/30-prior-checkpoint`

## What was wrong

`ContentView.emptySyncMessage` rendered "No new activities since [time]" using `SyncCheckpoint.load()`, but by the time the view re-rendered the checkpoint had already advanced. The result: a Sync at 6:52 PM following a 4:41 PM sync displayed "No new activities since 6:52 PM" instead of "since 4:41 PM" — the user wanted to know when the *previous* successful sync ran, not when the current one finished.

Race timing: in `SyncOrchestrator.syncActivities`'s `defer` block, the order was `SyncCheckpoint.save(finishedAt)` → `lastSyncFinishedAt = finishedAt`. SwiftUI observes the `lastSyncFinishedAt` flip (it's the `@Observable` property gating the empty-state branch) and re-renders, but by then the live checkpoint is already the just-completed sync's time.

Surfaced during Tom's manual verification of #24, the day #24 merged. The bug was always present in #24 — the test pack didn't catch it because there's no orchestrator-level integration test yet (#23 covers that).

## What changed

Two-file fix, no test infrastructure churn.

- **`SyncOrchestrator`** — adds `var priorCheckpoint: Date?`, an `@Observable` snapshot of `SyncCheckpoint.load()` taken at the *start* of `syncRecentActivities`, before the window is resolved. The orchestrator already held this value transiently in a `let`; promoting it to an observed property surfaces it for the UI.

- **`ContentView.emptySyncMessage`** — reads `sync.priorCheckpoint` instead of `SyncCheckpoint.load()`. The nil-fallback message ("No new activities.") stays as-is for the fresh-install case where there was no prior sync.

`syncSingleActivity` and `syncBackfill` don't touch `priorCheckpoint` — they don't advance the checkpoint either, and the empty-state UI doesn't fire after them (single-activity always has 1 item; backfill explicitly takes an `after` date and the UI doesn't reset session state on its return).

## Decisions made

- **Snapshot on the orchestrator vs. computed-property hack** — chose the snapshot. A computed property like "the previous-to-current checkpoint" would have required either a two-deep value history in `SyncCheckpoint` or reading the second-newest entry from `SyncLog` (which has its own success/failure mixing problems — see #24 post-mortem). A single in-memory snap on the orchestrator is one property, one assignment, zero new persistence.
- **Why not surface as a method like `func snapshotCheckpoint()`** — it'd be called from one place and nowhere else. YAGNI.
- **No new tests** — the change is "store this read in a property and surface it." Unit-testable behavior would be "did `priorCheckpoint` capture the right value before `syncActivities` was called?" which requires either a fake `StravaClient` (blocked on #23) or a contrived test that bypasses the orchestrator's actual sync flow. Tom's manual verification confirmed the fix; integration coverage rides on #23.

## What surfaced during verification

Clean. Sync twice, second tap showed the first sync's timestamp as expected. No other regressions noticed.
