# #5 — Phase 3: Background App Refresh + sync UI polish

## What was built

iOS BG App Refresh wires the existing Strava→HealthKit sync to run without Tom opening the app, with a 1-hour minimum interval (iOS picks the actual cadence). A persistent log captures the last 10 sync attempts (timestamp, source = foreground/background, activities processed, error info) and surfaces them in a "Recent syncs" section on the main screen. After 3 consecutive top-level sync failures, a local notification ("Sync failing — FitnessLoadTracker hasn't synced successfully in 3 attempts…") fires once per streak; a subsequent successful sync resets the counter.

Landed in one PR off branch `claude/5a-bg-refresh` covering three sub-slices in three commits (single PR per the new working-branch verification flow — see *Slicing pivot* below):
- **#5a** (`a1c4774` + diagnostic-readout `e55d465` + Info.plist file switch `8ef30fa`): BG refresh registration, scheduling, debug status readout.
- **#5b** (`6af3d4c`): `SyncLog` + `SyncLogEntry`, source attribution, Recent syncs UI.
- **#5c** (`b0be937`): `FailureNotifier` with pure `decision()` function, UNUserNotificationCenter dispatch, anti-spam flag.

Plus `081abe8` removing the now-unused "Set effort 7 on most recent workout" debug button (#4a-era scaffolding superseded by end-to-end sync).

## Decisions worth preserving

### Info.plist switched from `GENERATE_INFOPLIST_FILE` to a real file

The blocker that turned up during #5a's first on-device run: `BGTaskScheduler.register(forTaskWithIdentifier:...)` logged "Registration rejected; `dev.holmes.fitnessloadtracker.sync` is not advertised in the application's Info.plist." `plutil` on the built `.app/Info.plist` confirmed: neither `UIBackgroundModes` nor `BGTaskSchedulerPermittedIdentifiers` made it through, even though both `INFOPLIST_KEY_UIBackgroundModes = fetch;` and `INFOPLIST_KEY_BGTaskSchedulerPermittedIdentifiers = dev.holmes.fitnessloadtracker.sync;` were set in the pbxproj.

Xcode's `INFOPLIST_KEY_*` build-setting bridge only honors keys on Apple's documented allowlist. `BGTaskSchedulerPermittedIdentifiers` isn't on it; Xcode silently drops the value. Even for `UIBackgroundModes` (which seems like it should be bridged), the generator didn't produce a proper string-array — the runtime check rejected the malformed entry.

Resolution: ship a real `Info.plist` file. Settled at `ios/FitnessLoadTracker/Info.plist` (not under `ios/FitnessLoadTracker/FitnessLoadTracker/`). Reason for the directory choice: the source folder is a `PBXFileSystemSynchronizedRootGroup` that auto-includes all files in Copy Bundle Resources. Putting `Info.plist` there triggered "Multiple commands produce ... `Info.plist`" because Xcode treats the file both as the target's Info.plist (via `INFOPLIST_FILE`) AND as a bundle resource. Moving it one directory up avoids the synced group.

The file mirrors every key the generator was producing (CFBundle*, NSHealthShare/Update*, UIApplicationSceneManifest with the empty UISceneConfigurations, UILaunchScreen nested dict, UISupportedInterfaceOrientations~iphone/~ipad, UIRequiredDeviceCapabilities, UIApplicationSupportsIndirectInputEvents) — needed because `GENERATE_INFOPLIST_FILE = NO` means Xcode doesn't merge generator content with the file; the file is authoritative. Build-variable substitution (e.g., `$(PRODUCT_BUNDLE_IDENTIFIER)`) still works for the CF keys.

Build-time injected keys (BuildMachineOSBuild, DTPlatform*, DTSDK*, DTXcode*, MinimumOSVersion, UIDeviceFamily, CFBundleSupportedPlatforms) don't need to be in the file — Xcode adds them at build time regardless.

### Slicing pivot: single PR for three logical slices

Originally proposed three PRs matching the #4 precedent (4a/4b/4c). Tom realized mid-#5a that under the new working-branch verification flow (codified in CLAUDE.md during #12 close-out), there's no reason to split: verification happens on the branch before merge, so each slice's verification cycle is just additional commits on the same branch. Three PRs would add ceremony without value.

New pattern for this kind of multi-slice work: **separate commits per slice (the git log encodes the boundaries), single PR at the end.** Each commit's message names the slice (`(#5a)`, `(#5b)`, `(#5c)`) so the slicing intent is preserved.

This is the second issue closed under the new working-branch flow. Pattern is holding up.

### `BackgroundSync.handle(_:)` re-arms BEFORE running the sync

```swift
@MainActor
private static func handle(_ task: BGAppRefreshTask) {
    scheduleNext()  // ← BEFORE the work
    // ...spawn Task that runs sync, setTaskCompleted...
}
```

iOS gives a BG task ~30 seconds before forcibly expiring it. If sync is slow and hits the expiration handler, we still want a next-fire scheduled. Calling `scheduleNext()` up front is cheap insurance — `BGTaskScheduler.submit` is idempotent (replaces any pending request with the new one), so this can't double-schedule. Worst case is the BG sync expires AND fails to reschedule from inside the task; first case the new schedule from `handle`'s entry is still in place.

### `Task { @MainActor in ... }` bridge inside the BG handler

`BGTaskScheduler.register(...)` invokes its handler closure on a background thread. `SyncOrchestrator` is `@Observable` and — under Xcode 26's `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` (see `completed/4.md`) — its methods are implicitly MainActor. Bridging:

```swift
let work = Task { @MainActor in
    await sync.syncRecentActivities(source: .background, healthKit: healthKit)
    task.setTaskCompleted(success: sync.errorMessage == nil)
}
task.expirationHandler = { work.cancel() }
```

Same pattern any time a non-MainActor system callback needs to call into the project's `@Observable` types.

### Foreground vs background source is an explicit parameter, not inferred

`syncRecentActivities(source: SyncLogEntry.Source, ...)`. The foreground caller (`ContentView`'s Sync-now button) passes `.foreground`; `BackgroundSync.handle` passes `.background`. Considered alternatives:
- **Global flag set by BG entry point** — implicit, error-prone, hard to test.
- **Inferring from `Thread.isMainThread` / actor context** — fragile, doesn't survive refactors.

Explicit param wins: testable, refactor-safe, the call site states intent.

### Live BG status readout kept alongside Recent syncs

Initially planned to remove the debug readout (`BG: pending=…, next=…` + refresh status + last submit error) when #5b's Recent syncs UI lands. Reconsidered during #5b: the two views surface different information.

- **Recent syncs**: historical record of what happened.
- **BG status**: live state of the *next* scheduled fire and the system's per-app refresh permission.

Both useful. Removing the BG status would leave no in-app way to see (a) whether the BG task is currently registered, (b) the system permission state, (c) the last submit error if scheduling silently fails. Keep both; future cleanup can revisit once Tom is satisfied BG is rock-solid.

### `FailureNotifier.decision()` is a pure function

```swift
static func decision(log: [SyncLogEntry], defaults: UserDefaults) -> Decision
```

Returns `.fire(latestError:) | .resetStreak | .noop`. The side-effect-y `evaluate()` calls `decision()` then dispatches the notification + writes the flag. This lets the 7-test pack exercise every streak transition deterministically without touching UNUserNotificationCenter — same pattern as `Matching.findMatch` keeping HealthKit out of the test surface.

### Per-item errors don't count toward the failure streak

Failure = `errorSummary != nil` (OAuth refresh fail, network down, fetchActivities error). Per-item errors (no match, no score, no suffer score) are routine outcomes and don't indicate the sync run itself failed. Conflating them would notify Tom every time he had 3 rides in a row without HK matches — false-positive noise.

### Anti-spam via UserDefaults flag, not log inspection

Alternative: each sync inspects "have we ever notified for this exact streak boundary?" Hard to define precisely (what defines "this" streak vs. a new one?). Simpler:
- Flag = true after firing.
- Flag = false after any successful sync.
- Fire only when streak hit AND flag false.

Clean state machine, one bit of state.

### Notification permission ask at app launch

Considered just-in-time ask at the moment of the first failure-streak fire. Rejected: if the user declines at that moment (e.g., busy, distracted), the failure notification is lost AND the dialog is gone until they manually enable in iOS Settings. Asking at launch alongside HealthKit auth: user is in setup mindset, expects permission prompts, and grants once → all subsequent notification dispatches succeed.

If declined at launch, `UNUserNotificationCenter.add(_:)` silently no-ops. Failures still get logged in Recent syncs; only the push surface is muted. Tom can flip the iOS Settings toggle to restore.
