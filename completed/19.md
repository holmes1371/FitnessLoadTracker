# #19 — Custom app icon (cyclist on jersey-blue)

## What was built

Replaced the placeholder app icon with a custom one: filled black cyclist silhouette in a road-bike racing tuck on a `#1E5FBF` jersey-blue background. The icon is rendered programmatically from `ios/scripts/generate-app-icon.swift` (CoreGraphics + ImageIO) — run with `swift ios/scripts/generate-app-icon.swift <output-path>`. The output PNG (`Icon-1024.png`) is committed into the asset catalog and wired to the "any" appearance only; dark and tinted variants are deferred, and Apple falls back to "any" when those aren't supplied.

Landed in three commits on branch `claude/bold-feynman-9d1120`:
- `926213e`: first pass — recognizable cyclist + bike, but the rider's legs merged with the seat tube and arms were stubby.
- `0535119`: lengthened arms (~140px → ~235px) by pulling the shoulder back from `(660, 360)` to `(590, 350)`, moved the visible pedal to a forward-down power-stroke position, and added the blue-halo trick for leg/frame separation.
- `51ee78d`: 60px upward CTM shift to vertically center the composition on the canvas.

## Decisions worth preserving

### Script as source of truth, PNG committed as build product

The script is deterministic — re-running produces a bit-for-bit identical PNG. The committed PNG is a build product, not the canonical artifact. Future tweaks edit the geometry constants in the script and re-run it; the PR diff shows the new PNG alongside the geometry change so the visual delta is reviewable on GitHub.

Generating the PNG at build time was rejected — the asset changes rarely, and a build-time render would have added a Swift script invocation to every clean build for no real benefit. The script-emits-PNG-checked-in approach is the same shape as `ios/scripts/generate-secrets.sh` for the secrets file, just with a different output type.

### Blue-halo overstroke for silhouette/frame separation

First pass painted the rider in pure black on top of a pure-black bike. The rider's upper leg sat flush along the seat tube — same slope, same color — and the whole rider-on-bike region collapsed into one indistinct black blob. The legs were drawn but visually invisible.

Fix: before drawing each leg segment in black, draw the same segment in background-blue with a 12px-wider stroke. The halo carves a visible blue gap through any frame tube the leg crosses, so the leg reads as clearly in front of the bike. Implementation in `generate-app-icon.swift`:

```swift
ctx.setStrokeColor(bgBlue)
stroke({ p in p.move(to: hip);  p.addLine(to: knee) },  width: 60)  // halo
stroke({ p in p.move(to: knee); p.addLine(to: pedal) }, width: 44)
ctx.setStrokeColor(black)
stroke({ p in p.move(to: hip);  p.addLine(to: knee) },  width: 48)  // leg
stroke({ p in p.move(to: knee); p.addLine(to: pedal) }, width: 32)
```

Reusable for any future flat asset where one black mass needs to read clearly in front of another. The technique only works when the foreground shape is drawn last (so its halo overstrokes the background) — drawing order matters.

### Drawing order matters: torso covers leg halo at hip

The leg halo at the hip end would, on its own, carve a blue ring around the rider's hip — visible as a gap between the rider's bum and the saddle/seat tube. That's not what we want; the rider sits on the saddle and the hip-saddle junction should read continuous.

Resolution: draw legs BEFORE torso. The torso line ends in a round cap at the hip (width 72, half-radius 36) which fully covers the small leg halo (half-radius 30) at that endpoint. The halo at the knee bend (mid-leg) and pedal (foot) stays visible against the frame, which is what's needed.

If geometry shifts or drawing order changes, expect the hip-side halo to reappear. Worth checking visually after any non-trivial geometry edit.

### CTM translation for composition centering, not per-coordinate offset

When the composition came out 77px below canvas center, the centering fix was one line: `ctx.translateBy(x: 0, y: -60)` after the background fill, before any bike/rider drawing. The alternative — adding `-60` to every y-coordinate in the geometry section — would have polluted the constants with a presentation-layer concern. Keep geometry coordinates as "where this part sits in canvas space conceptually," and apply layout shifts via the CTM.

### Side-view racing tuck, not three-quarter or front

Three-quarter and front views are harder to read at home-screen size with a single-color silhouette — the bike loses its iconic profile and the rider becomes a featureless blob. Side view gives the recognizable double-wheel-and-frame-triangle shape plus a clearly readable rider posture. Racing tuck (bent torso, head forward, arms reaching to drop bars) reinforces "this is a cycling app" rather than "this is a generic person on a bicycle."

### Single "any" appearance only

Apple's icon system supports three appearance variants (any / dark / tinted). The asset catalog was already configured for all three slots, but only "any" gets a PNG; the others stay empty and Apple falls back to "any." Generating dark/tinted variants at this stage would have been speculative — Tom hasn't asked for them, and adding them is non-breaking later. Skipping them keeps the script + asset catalog smaller now.

## Gotchas

- **`ios/.env` symlink per worktree.** First build in a fresh worktree fails with `Missing .../ios/.env` because `generate-secrets.sh` runs as a pre-Sources build phase and needs the secrets file. The fix in this worktree was `ln -s ../../../../ios/.env ios/.env` — relative target so it survives if Tom ever moves the project. The symlink is gitignored (`*.env` in `.gitignore`) and stays local to the worktree.
- **`swift <file.swift>` CLI mode is enough.** No Xcode project or `Package.swift` needed for a one-off script that uses CoreGraphics, ImageIO, and UniformTypeIdentifiers — they're available out of the box on macOS. Useful pattern for any future deterministic asset generation.
- **Lower arm crosses bike stem.** With the current geometry, the rider's lower arm passes over the bike's stem on its way to the bar drops. Drawing the rider after the frame hides this crossing naturally (the arm covers the part of the stem behind it). Geometry shifts could expose this — if you see a weird notch in the stem area, this is why.

## What's NOT in here

- No XCTest target tests. The script's "test" is the visual confirmation on device — there's nothing meaningful to assert programmatically about a 1024×1024 PNG short of pixel-checksum tests, which would just lock in the current geometry as if it were correct.
- No dark / tinted appearance variants. Deferred.
- No alternate icons (e.g., a "pro" or seasonal alternate). Out of scope.

## Future tweaks

The geometry is all named constants at the top of `generate-app-icon.swift`. To tweak the icon: edit constants, re-run `swift ios/scripts/generate-app-icon.swift <path>`, eyeball the result, commit. Likely candidates for future iteration if Tom wants them:
- Helmet on the rider's head (currently bare).
- A second visible leg in the back half of the pedal stroke.
- Dark / tinted appearance variants (each is its own `Icon-1024-<variant>.png` plus the matching `filename` entry in `Contents.json`).
