# #35 — BG sync fails with "Protected health data is inaccessible" when phone is locked

**Closed:** 2026-05-28
**Branch:** `claude/35-protected-data-guard`

## What was wrong

Background syncs intermittently logged a ⚠ row reading "Protected health data is inaccessible" — surfaced directly thanks to #32's `firstItemError` instrumentation. The string is `HKError.errorDatabaseInaccessible`: HealthKit is encrypted at rest and unreadable while the device is locked. A `BGAppRefreshTask` can fire whether or not the phone is unlocked, so when iOS woke the app for a sync on a locked phone, every HK read/write in `SyncOrchestrator.process()` threw and each fetched activity landed as a per-item `.error`. Purely lock-state-dependent at fire time — hence intermittent (in the reporting screenshot, the 11:51 AM fire succeeded while 10:21 AM / 12:52 PM / 1:53 PM failed). Foreground "Sync now" was never affected (phone unlocked when tapped).

## The latent bug behind the noise

The more dangerous half: the checkpoint advance only inspects the top-level `errorMessage`:

```swift
if errorMessage == nil { SyncCheckpoint.save(finishedAt) }
```

Protected-data failures surface as *per-item* errors, which never set `errorMessage`. So a locked-device fire failed every activity yet still advanced `SyncCheckpoint` as if they'd been processed. The 24h overlap and manual syncs covered for it most of the time, but an activity only ever "seen" during locked fires could be silently skipped.

## What changed

One guard at the top of `BackgroundSync.handle()` (already `@MainActor`, already re-arms the next request first):

```swift
guard UIApplication.shared.isProtectedDataAvailable else {
    task.setTaskCompleted(success: true)   // not a failure — come back later
    return
}
```

Plus `import UIKit`. A locked fire now becomes a clean no-op: no Strava call we can't use, no noisy ⚠ row, and no checkpoint advance. iOS keeps firing; the work lands on the next fire that happens while unlocked (or the next manual sync).

## Decisions made and rejected alternatives

- **Guard in `BackgroundSync` vs error-classification in `SyncOrchestrator`.** The alternative was to detect `HKError.errorDatabaseInaccessible` inside `process()`'s catch and promote it to a top-level `errorMessage` (which already blocks the checkpoint and marks the BG task unsuccessful). Rejected as the primary fix because (a) it still wastes the Strava token-refresh + activity fetch on a locked device, and (b) promoting to a top-level error would feed the `FailureNotifier` 3-strike streak and push Tom a "Sync failing" notification for what's really just "phone was locked." The guard avoids all of that by not running at all.

- **`task.setTaskCompleted(success: true)`, not `false`.** A locked fire isn't a failure — it's a deferral. Reporting `false` could make iOS deprioritize future BG scheduling for the app, which is the opposite of what we want.

- **Kept the orchestrator UIKit-free.** The protected-data check is a BG-execution-environment concern, so it lives in `BackgroundSync` (which now imports UIKit) rather than the orchestrator (Foundation/HealthKit/Observation only). Foreground syncs are always unlocked, so they need no such check.

- **Did not harden the checkpoint-advance logic against environmental per-item errors.** The guard fixes the common case (locked at fire time → no run → no checkpoint advance). The residual gap is the phone locking *mid-sync* after the guard passes; it's rare, self-heals via the 24h overlap, and #32's per-item surfacing remains the safety net. The deeper "don't advance the checkpoint when per-item errors are environmental" fix is orchestrator territory and pairs naturally with the #23 integration-test work — kept out of scope here.

## What surfaced during verification

Tom chose to ship on trust and watch BG behavior over the next couple of days, rather than force-fire locked syncs from Xcode. The fix is one guard with no test-suite impact (full suite green), and the real signal is the absence of "Protected health data is inaccessible" rows in the wild. Manual-verification checkboxes on #35 stay unticked at close — they get checked as the days of clean BG rows accumulate.

## Testing note

`BackgroundSync` has no unit tests — it's all `BGTaskScheduler` / `UIApplication` glue, and a one-line lock-state guard isn't meaningfully testable without mocking `UIApplication`. Consistent with the existing untested BG surface; no mock infra built for it. The full XCTest pack still passes (the change touches only BG-handler control flow).
