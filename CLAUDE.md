## Session discipline

- **Plan before coding** — for any non-trivial task, write the plan out loud (what you'll change, which files, in what order, what you considered and ruled out). Wait for explicit user approval. Stating a plan and immediately executing it in the same turn is a violation of the karpathy-guidelines rule. "Earlier approval carries forward" applies only to mechanical continuations.
- **Design notes for non-trivial features** — `design/{feature-name}.md`. Captures scope, decisions already made, test fixtures needed. A fresh session should be able to pick up mid-feature from the design note plus the last commit, without re-litigating choices.
- **Commit at every natural boundary**, not just at feature completion. Half-finished work behind a clear commit message is recoverable; a dirty worktree is not.
- **Status flips on issues** — when the user approves a plan, move the issue from the project board's Todo column to **In Progress**. The board column is the only signal needed; do **not** add an `in-progress` label — it's redundant with the column and clutters the issue header. When work merges to main (all sub-checkboxes in the issue body checked, automated verification passing), move the issue to **In Testing** — the issue stays open. Do **not** move to Done and do **not** close on your own; Tom runs the manual-verification checklist and either closes the issue himself (typically via "Closes #N" in the closing commit message — which auto-moves it to Done) or returns feedback. If feedback comes back, move the issue back to In Progress and resume work.
- **PR timing — wait for Tom's explicit "ready for testing" before opening one.** When implementation on a working branch is complete, push the branch and summarize what landed — but **stop before opening a PR**. Tom reviews the working branch first and only says "raise the PR" when he wants it shipped for testing. Opening a PR proactively forecloses cheap iteration: any tweak Tom requests then has to land on a *fresh* branch + *fresh* PR (because the original is already merged or queued), instead of a follow-up commit on the existing working branch. Until Tom says "ready," keep iterating with additional commits on the same branch — that's the cheap loop. Failure mode: agent finishes implementation → opens PR reflexively → Tom requests a tweak → tweak lands on a new branch because the merged PR is gone.
- **No close-keywords in shipping PR bodies** — `Closes #N` / `Fixes #N` / `Resolves #N` (and their variants) anywhere in a PR body or in any commit included in it auto-closes the linked issue when GitHub merges the PR. That bypasses the In Testing → Tom's manual verification → close-out flow. Use `Refs #N` in the PR body instead; `(#N)` in the PR title is also safe (parenthetical references don't trigger auto-close). The close-keyword belongs only in the closing commit Tom writes (or the agent writes on his explicit say-so) **after** verification passes — that's when the issue actually transitions to Done. Failure mode: "the PR really does fix the issue, so 'Fixes #N' is accurate" — accurate is the wrong test; this is about *who* triggers close.
- **Labels at issue file time** — only `bug` if it's a bug. Otherwise no label; sitting in the Todo column is the convention. Do **not** add `in-progress` when work starts — the board's In Progress column already tracks that, and a redundant label clutters the issue header. Don't reach for other labels speculatively.
- **Manual verification checklist on issues** — when an item ships work that needs Tom's manual verification before close (running a CLI command and inspecting output, clicking through a UI flow, confirming a side effect in an external service), append a `## Manual verification` section to the issue body with checkbox items. Each box names one concrete check Tom will perform. Sub-task checkboxes in the scope section track *what was built* (agent ticks these as it builds); manual-verification checkboxes track *what Tom confirmed working* — **only Tom ticks the manual-verification boxes**, never the agent. They're his audit trail; ticking them on his behalf defeats the purpose. The issue is ready to close only when both sets are checked. Items that need no manual check (pure refactors, internal helpers verified by tests alone) skip the section.
- **Editing GitHub issue bodies via `gh`** — `gh issue edit --body "..."` **overwrites the entire body**. Tom may tick manual-verification boxes in the GitHub UI between the agent's edits; pushing a stale local copy back silently un-ticks them. Before each edit: `gh issue view <N> --json body --jq '.body' > /tmp/body.md`, modify only the sections you intend to change (preserve `## Manual verification` content verbatim from the fetched copy), then push back with `--body-file /tmp/body.md`. The agent **never** writes `[x]` in a manual-verification checkbox itself — only `[ ]` on initial creation, or whatever the fetched body shows on round-trips. The "I'm just preserving what I think is the current state" rationalization is the failure mode; always fetch.
- **Session-log file per session** — write one file at `.claude/session-log/YYYY-MM-DDTHHMM-<branch-slug>.md` (e.g. `2026-05-24T0900-framework-bootstrap.md`) holding ≤5 bullets, ≤1 sentence each, of cold-pickup content (current/most-recent session, kept short for cold pickup; trim ruthlessly — if a bullet needs a paragraph, the real content belongs in a design note, a `completed/*.md` post-mortem, or a GitHub issue). One file per session, rewritten in place between commits within that session (new session → new file, not new file per commit). Per-file design avoids the merge conflicts a single-block-in-CLAUDE.md convention produces when two agents work the same day on different branches. Naming: ISO date + 24-hour `HHMM` so files sort newest-last alphabetically; branch slug disambiguates parallel writes. Trim at session start: if `.claude/session-log/` has more than ~10 files or any file is older than ~14 days, delete the older ones — durable signal lives in commit messages, `completed/*.md` post-mortems, design notes, and closed issues; the session-log is recency-only.
- **Closed items archive to `completed/<NN>.md` immediately on close-out** — when the user signs off a closed issue, write the full post-mortem prose into `completed/<issue-number>.md` (e.g. `completed/12.md`) **right then**, in the same turn the issue gets closed. Not "before the session ends." Not "after I finish the next slice." The current turn has full context on decisions, bugs, and trade-offs that don't survive cleanly into commits / issue bodies / design notes alone; even waiting until end-of-session loses signal because by then you've moved through several issues and the per-issue detail blurs together. The right cadence: close the issue → write the post-mortem file → commit `completed/<NN>.md` → continue. Searchable repo-local history beats navigating closed issues. Per-file design (one file per issue, named by issue number) means parallel agents closing different issues the same day don't collide on shared lines. No trim discipline: `completed/` is an archive, files accumulate forever.
- **Landing the post-mortem commit on main** — the post-mortem commit (`completed/<NN>.md`) and the paired session-log file write at `.claude/session-log/` go directly on `main` in the parent worktree (path: `C:/DevWork/FitnessLoadSync`). From a feature worktree, operate on the parent via `git -C <parent path> ...` since `git checkout main` is locked to the parent. Verify the parent is clean first with `git -C <parent> status --porcelain` (untracked entries inside `.claude/` other than `session-log/` — e.g. harness state — are fine to ignore; abort on any tracked-file changes, including unstaged edits to existing `.claude/session-log/*.md` or `completed/*.md` files, since Tom may have in-progress work there). Edit, stage, commit on main locally — then **stop**. Don't push, don't open a PR. Tom pushes manually on his own cadence; the agent reports the local commit SHA. Use `Refs #N` in the commit message body, never `Closes #N` — by the time the post-mortem lands the issue is already closed, and `Closes` keywords on doc commits muddy issue searches. Scope: this bypass is for the post-mortem `completed/<NN>.md` file and the paired session-log file only — code, design notes, and CLAUDE.md edits that change *behavior* still go through the normal feature-branch + PR + CI flow.
- **Tests live in `tests/`** and (once CI is wired up) run on every push + PR via GitHub Actions. A red CI check blocks merge; do not mark a feature done with tests failing.
- **External-service walkthroughs**: when an issue requires the user to set up an external service (Garmin / Strava / Apple Health / Google Cloud / OAuth apps, etc.), the issue body or its linked design note must include a step-by-step walkthrough — every screen the user will see, every value to paste, every secret to copy where. The user is doing this from scratch and a missed step blocks the whole item. When a step requires a value only the user can produce (account email, payment method, OAuth client ID), call it out explicitly and pause until they confirm.
