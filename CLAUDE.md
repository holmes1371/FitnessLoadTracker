## Session discipline

- **Plan before coding** — for any non-trivial task, write the plan out loud (what you'll change, which files, in what order, what you considered and ruled out). Wait for explicit user approval. Stating a plan and immediately executing it in the same turn is a violation of the karpathy-guidelines rule. "Earlier approval carries forward" applies only to mechanical continuations.
- **Design notes for non-trivial features** — `design/{feature-name}.md`. Captures scope, decisions already made, test fixtures needed. A fresh session should be able to pick up mid-feature from the design note plus the last commit, without re-litigating choices.
- **Commit at every natural boundary**, not just at feature completion. Half-finished work behind a clear commit message is recoverable; a dirty worktree is not.
- **Status flips on issues** — when the user approves a plan, move the issue from the project board's Todo column to **In Progress**. The board column is the only signal needed; do **not** add an `in-progress` label — it's redundant with the column and clutters the issue header. Manual verification happens on the working branch *before* the PR opens (see *Verification on the working branch* and *PR timing* below); the issue stays In Progress through that phase. When the PR merges with `Closes #N` in its body, GitHub auto-closes the issue and the board moves to **Done**. The "In Testing" column is legacy from a previous post-merge-verification flow and is unused under the working-branch model — leave it on the board but don't route issues through it.
- **Verification on the working branch** — for iOS work, manual verification happens *before* the PR opens, against the agent's feature-branch worktree directly. Tom opens the worktree's `ios/FitnessLoadTracker/FitnessLoadTracker.xcodeproj` in Xcode (not the parent worktree's) — the worktree path is enough; no branch switching in the parent is needed. The `ios/.env` is symlinked from the parent worktree on first build so secrets generation works. He builds, deploys to his phone, runs through the issue's `## Manual verification` checklist, and ticks the boxes himself in the GitHub UI. If feedback comes back, the agent commits the fix to the same working branch — cheap loop, no new branch needed. When the checklist is complete (or Tom explicitly waives it on a low-risk change he's content to ship on test-suite trust), the close-out artifacts (`completed/<NN>.md` post-mortem, session-log file) get bundled onto the same branch and the PR opens for merge.
- **PR timing — wait for Tom's explicit "raise the PR" / "ready to merge" before opening one.** When implementation on a working branch is complete, push the branch and summarize what landed — but **stop before opening a PR**. Tom verifies on the working branch first (see *Verification on the working branch* above) and only signals to open the PR when verification is done (or he chooses to merge on trust). Opening a PR proactively forecloses cheap iteration: any tweak Tom requests then has to land on a *fresh* branch + *fresh* PR (because the original is already merged or queued), instead of a follow-up commit on the existing working branch. Until Tom says "raise it," keep iterating with additional commits on the same branch — that's the cheap loop. Failure mode: agent finishes implementation → opens PR reflexively → Tom requests a tweak → tweak lands on a new branch because the merged PR is gone.
- **Close-keywords in PR bodies are allowed (and expected)** — `Closes #N` / `Fixes #N` / `Resolves #N` in a PR body auto-closes the linked issue when GitHub merges the PR. Under the working-branch verification model, this is the intended trigger: merge happens *after* manual verification is complete (or waived), so merge = done = close. Put `Closes #N` in the PR body when verification has been ticked off or Tom has approved the merge. Use `Refs #N` instead when the PR is incremental work that doesn't yet finish the issue — e.g., a multi-PR slice where earlier slices land before the issue is fully resolved.
- **Labels at issue file time** — only `bug` if it's a bug. Otherwise no label; sitting in the Todo column is the convention. Do **not** add `in-progress` when work starts — the board's In Progress column already tracks that, and a redundant label clutters the issue header. Don't reach for other labels speculatively.
- **Manual verification checklist on issues** — when an item ships work that needs Tom's manual verification before close (running a CLI command and inspecting output, clicking through a UI flow, confirming a side effect in an external service), append a `## Manual verification` section to the issue body with checkbox items. Each box names one concrete check Tom will perform. Sub-task checkboxes in the scope section track *what was built* (agent ticks these as it builds); manual-verification checkboxes track *what Tom confirmed working* — **only Tom ticks the manual-verification boxes**, never the agent. They're his audit trail; ticking them on his behalf defeats the purpose. The issue is ready to close only when both sets are checked. Items that need no manual check (pure refactors, internal helpers verified by tests alone) skip the section.
- **Editing GitHub issue bodies via `gh`** — `gh issue edit --body "..."` **overwrites the entire body**. Tom may tick manual-verification boxes in the GitHub UI between the agent's edits; pushing a stale local copy back silently un-ticks them. Before each edit: `gh issue view <N> --json body --jq '.body' > /tmp/body.md`, modify only the sections you intend to change (preserve `## Manual verification` content verbatim from the fetched copy), then push back with `--body-file /tmp/body.md`. The agent **never** writes `[x]` in a manual-verification checkbox itself — only `[ ]` on initial creation, or whatever the fetched body shows on round-trips. The "I'm just preserving what I think is the current state" rationalization is the failure mode; always fetch.
- **Session-log file per session** — write one file at `.claude/session-log/YYYY-MM-DDTHHMM-<branch-slug>.md` (e.g. `2026-05-24T0900-framework-bootstrap.md`) holding ≤5 bullets, ≤1 sentence each, of cold-pickup content (current/most-recent session, kept short for cold pickup; trim ruthlessly — if a bullet needs a paragraph, the real content belongs in a design note, a `completed/*.md` post-mortem, or a GitHub issue). One file per session, rewritten in place between commits within that session (new session → new file, not new file per commit). Per-file design avoids merge conflicts when parallel agents work different branches the same day. Naming: ISO date + 24-hour `HHMM` so files sort newest-last alphabetically; branch slug disambiguates parallel writes. The file lands on the feature branch (rides the closing PR into main) under the standard flow; on parent-worktree main directly for sessions that don't produce a PR (see *Sessions that don't produce a feature PR* below). Trim at session start: if `.claude/session-log/` has more than ~10 files or any file is older than ~14 days, delete the older ones — durable signal lives in commit messages, `completed/*.md` post-mortems, design notes, and closed issues; the session-log is recency-only.
- **Closed items archive to `completed/<NN>.md` before the closing PR opens** — when Tom signals "ready to merge," write the full post-mortem prose into `completed/<issue-number>.md` (e.g. `completed/12.md`) **right then, on the feature branch, before opening the PR**. The current turn has full context on decisions, bugs, and trade-offs that don't survive cleanly into commits / issue bodies / design notes alone; even waiting until "after I open the PR" loses signal as the conversation moves on. The right cadence: Tom says raise the PR → agent writes post-mortem + session-log on the working branch → commits both → opens the PR with `Closes #N` in the body → PR merges → issue closes → `completed/<NN>.md` lands on main with the rest of the feature. Searchable repo-local history beats navigating closed issues. Per-file design (one file per issue, named by issue number) means parallel agents closing different issues the same day don't collide on shared lines. No trim discipline: `completed/` is an archive, files accumulate forever.
- **Sessions that don't produce a feature PR** — for research-only sessions, planning-only sessions, or sessions that pause mid-feature without merging, write the session-log file on the **parent worktree's main** (path: `/Users/tomholmes/DevWork/FitnessLoadTracker/FitnessLoadTracker/`) so it's discoverable to the next session that branches from main. From a feature worktree, use `git -C <parent path> ...` since `git checkout main` is locked to the parent. Verify the parent is clean first with `git -C <parent path> status --porcelain` (untracked entries inside `.claude/` other than `session-log/` — e.g. harness state — are fine; abort on any tracked-file changes, since Tom may have in-progress work there). Stage and commit locally on main — don't push. Tom pushes on his own cadence. Use `Refs #N` in any post-mortem committed this way, never `Closes #N` on a doc-only commit. Under the standard flow (verification on working branch + feature PR), the session-log just rides the feature branch — no parent-worktree bypass needed.
- **Tests live alongside the code they exercise — a regression pack that accumulates with each feature.** Swift unit tests in the Xcode XCTest target at `ios/FitnessLoadTracker/FitnessLoadTrackerTests/` — run locally with **⌘U**. Shell-script tests in `<dir>/tests/` next to the script (e.g. `ios/scripts/tests/test-generate-secrets.sh` — run with `bash <path>`, exits non-zero on any failure). Every feature adds regression tests for behavior it introduces; agents do NOT smoke-test inline as a substitute. Once CI is wired up, both run on every push + PR; a red CI check blocks merge; do not mark a feature done with tests failing.
- **External-service walkthroughs**: when an issue requires the user to set up an external service (Garmin / Strava / Apple Health / Google Cloud / OAuth apps, etc.), the issue body or its linked design note must include a step-by-step walkthrough — every screen the user will see, every value to paste, every secret to copy where. The user is doing this from scratch and a missed step blocks the whole item. When a step requires a value only the user can produce (account email, payment method, OAuth client ID), call it out explicitly and pause until they confirm.
